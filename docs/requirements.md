# vLLM Ascend 社区看板项目需求文档

## 1. 项目概述

### 1.1 项目背景

vLLM Ascend 是一个针对华为昇腾（Ascend）AI 处理器优化的 vLLM 项目。为了更好地展示项目进展、CI 状态、模型看板情况和性能指标，需要建设一个社区看板，为开发者、贡献者、终端用户和管理层提供统一的信息汇聚和展示平台。

### 1.2 项目目标

- 提供 vLLM Ascend 项目 CI 状态的可视化展示（夜间测试、性能测试）
- 展示测试覆盖率和测试通过率等质量指标
- 提供动态管理的各大模型功能验证报告
- 展示各模型在 Atlas A2/A3 硬件上的性能指标看板
- 支持历史数据追溯和趋势分析

### 1.3 目标用户

| 用户类型 | 使用场景 |
|---------|---------|
| 开发者 | 查看 CI 状态、测试失败原因、性能回归 |
| 贡献者 | 了解 PR 合入门槛、测试要求 |
| 终端用户 | 评估模型支持情况、性能表现 |
| 管理层 | 项目整体进度、质量趋势 |

## 2. 功能需求

### 2.1 CI 看板模块

#### 2.1.1 CI 状态展示

- **实时状态**：展示当前 CI 流水线运行状态（成功/失败/进行中）
- **历史趋势**：CI 通过率随时间变化的趋势图
- **运行分析**：
  - CI 运行时长统计
  - 失败原因分类分析
  - 各 Job 状态详情

#### 2.1.2 测试类型覆盖

| 测试类型 | 说明 | 数据源 |
|---------|------|--------|
| 夜间测试 (Nightly) | 每日定时运行的单节点/多节点 E2E 测试 | GitHub Actions `schedule_nightly_test_*` |
| 性能测试 (Benchmarks) | 定期运行的延迟/吞吐量/服务性能测试 | GitHub Actions `schedule_test_benchmarks` |
| 文档测试 (Doctests) | 文档示例代码测试 | GitHub Actions `_e2e_test.yaml` |

#### 2.1.3 测试指标

- **测试通过率**：各测试套件通过率（按硬件 A2/A3 分类）
- **测试覆盖率**：代码覆盖率统计及趋势（Codecov）
- **可配置 Job**：支持配置需要监控的 GitHub Actions Job

#### 2.1.4 数据更新

- 定时轮询 GitHub API（默认每 10 分钟）
- 支持手动刷新
- 更新频率可配置

### 2.2 模型看板报告模块

#### 2.2.1 模型管理

- **动态管理**：支持管理员对模型进行增删改查
- **模型分类**：支持按模型系列分类（Qwen、Llama、DeepSeek、其他）
- **测试配置**：基于 `tests/e2e/models/configs/*.yaml` 格式管理模型测试配置

#### 2.2.2 验证报告内容

| 报告项 | 说明 | 数据来源 |
|-------|------|---------|
| Pass/Fail 状态 | 模型功能验证是否通过 | lm_eval 测试结果 |
| 特性列表 | 支持的算子/功能清单 | 测试配置 |
| 已知问题 | 当前存在的问题和限制 | 管理员录入 |
| 质量检查 | 输出质量与基线的对比分析 | exact_match 等指标 |

**Pass/Fail 判定标准**：
- 所有测试任务的准确率指标均达到配置阈值
- 任一任务未达标则判定为 Fail
- 支持管理员手动调整判定结果

#### 2.2.3 报告展示

- **解析原始数据**：从 GitHub Actions artifacts 获取 lm_eval 原始 JSON 结果
- **解析 Markdown 报告**：基于 `report_template.md` 模板生成的报告
- **链接原始报告**：提供原始报告文件的直接访问链接

### 2.3 性能看板模块

#### 2.3.1 性能指标

- **吞吐量**：Tokens/sec、Requests/sec
- **延迟**：首 Token 时间 (TTFT)、端到端延迟、Token 间延迟 (ITL)
- **显存占用**：GPU 内存消耗

#### 2.3.2 测试类型

| 测试类型 | 说明 | 指标 |
|---------|------|------|
| 延迟测试 | 固定输入 32 tokens，输出 128 tokens，batch_size=8 | 端到端延迟（均值、中位数、P99） |
| 吞吐量测试 | ShareGPT 200 prompts，动态 batch size | Throughput |
| 服务测试 | ShareGPT 200 prompts，QPS=1/4/16/inf | Throughput、TTFT、ITL |

#### 2.3.3 硬件维度

- **Atlas A2**：910B1 芯片
- **Atlas A3**：新一代芯片

#### 2.3.4 基准对比

- **版本对比**：支持与历史任意 vLLM 版本对比（如 v0.16.0）
- **日期对比**：支持与历史任意日期数据对比
- **Commit 对比**：支持按 Commit 查询和对比
- **目标值对比**：支持手动输入目标值进行对比

#### 2.3.5 数据来源

- **自动采集**：从 GitHub Actions artifacts 自动获取性能数据（JSON 格式）
- **手动上传**：支持管理员按标准 YAML/JSON 格式手动上传

### 2.4 用户认证与权限模块

#### 2.4.1 认证方式

- 简单用户名/密码登录
- 会话管理

#### 2.4.2 权限分级

| 角色 | 权限 |
|-----|------|
| 游客 | 公开内容只读 |
| 登录用户 | 查看需登录模块 |
| 普通管理员 | 模型管理、数据上传、配置修改 |
| 超级管理员 | 用户管理、系统配置、权限分配 |

#### 2.4.3 访问控制

- 默认内容公开可读
- 内容编辑需管理员权限
- 个别模块需登录后可见

### 2.5 历史数据模块

#### 2.5.1 数据存储

- 存储所有 CI、性能、验证报告的历史数据
- **数据保留策略**：默认永久保留，支持配置清理策略

#### 2.5.2 趋势展示

- 时间序列图表展示指标变化趋势（天/周/月）
- 支持自定义时间范围查询
- 支持按 vLLM 版本筛选对比
- 支持按 Commit 查询详细结果

## 7. 异常处理与容错

### 7.1 GitHub API 限流处理

| 限流类型 | 限制 | 处理策略 |
|---------|------|---------|
| 未认证请求 | 60 次/小时 | 使用认证 Token，提升至 5000 次/小时 |
| 认证请求 | 5000 次/小时 | 请求头携带 `X-RateLimit-Remaining` 监控 |
| 二次限流 | 动态 | 指数退避重试（1min, 5min, 15min） |

- **降级方案**：限流时显示缓存数据，标注数据更新时间
- **告警机制**：连续 3 次触发限流时记录日志

### 7.2 数据解析失败处理

| 场景 | 处理策略 |
|-----|---------|
| Artifacts 格式不兼容 | 记录错误，跳过该条数据，继续处理其他数据 |
| JSON/YAML 解析失败 | 保留原始数据，标记为"解析失败"，支持手动修复 |
| 必填字段缺失 | 使用默认值或标记为"数据不完整" |
| 报告模板变更 | 支持多版本模板解析，无法解析时链接到原始文件 |

### 7.3 网络异常处理

- **重试机制**：
  - 初始重试：立即重试 1 次
  - 短期重试：间隔 30 秒，最多 3 次
  - 长期重试：间隔 5 分钟，最多 3 次
- **超时设置**：
  - GitHub API 请求：30 秒
  - Artifacts 下载：5 分钟
  - 数据库操作：10 秒

### 7.4 数据一致性保障

- **幂等性**：数据采集支持重复执行，自动去重
- **事务性**：关键数据写入使用数据库事务
- **回滚机制**：批量数据处理失败时回滚到上一致状态

## 9. 安全性需求

### 9.1 认证安全

- **密码存储**：使用 bcrypt/argon2 加密存储
- **会话管理**：JWT Token，有效期 24 小时，支持刷新
- **登录保护**：连续 5 次失败后锁定 30 分钟

### 9.2 数据安全

- **敏感信息**：GitHub Token 等敏感信息加密存储
- **权限校验**：所有写操作接口进行权限验证
- **SQL 注入防护**：使用 ORM 或参数化查询

### 9.3 接口安全

- **速率限制**：API 接口限流（100 次/分钟/IP）
- **输入验证**：所有输入参数进行格式校验
- **CORS 配置**：限制允许的源地址
